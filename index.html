Google_IO2015_Brillo_Weave

Posted on 2015-06-01   |   In 业内动态   |  

Brillo是Google打算进军智能硬件和家庭自动化的棋子。Google主要是从Nest的团队中和Android的团队中组了一个新团队，来进行Brillo的研发。Brillo可以被看做是一个主要为了小型嵌入式设备而做的Android系统，比如各种家电控制器、路由器、传感器控制器等等。

而Weave是一种智能硬件的通信语言，它本来的字面意思是编织，很明显，Google想用它来达到万物相连像织布一样的效果。

目前IOT行业有几个大问题。第一是缺乏标准，不仅是大公司都想着推出自己的标准来垄断市场，小公司也都按照自己的想法来实现产品，这样出的问题就是大家的产品都只能在自家的阵营里玩。出了这个圈子，就没法跟其他产品兼容。标准化的工作一般动辄都是以十数年的工作才能完成，不仅仅是技术问题，也是各种商业利益博弈。而Google和Apple这类巨头，都试图主导行业的标准化，这样就可以保持行业的领先地位。第二个是目前的IOT和智能家居还是离大众化太远了，缺少真正强大的市场需求，目前技术所能做到的仍然大部分只是用户对家电设备进行最基本的操控，比如开关，调节之类，还只能算是半需求半卖概念的状态。

Google推出Brillo和Weave的目的就在于试图统一IOT行业的底层，让创业公司专心去发掘终端应用。一方面操作系统的统一可以让各硬件和OEM厂商可以像现在的Android一样大量出货降低成本，另一方面软件上基于Json的Weave本身就具备了完美支持手机应用和Web的属性。这样可以避免类似于Phillips Hue里面的Hub那样的中间件，直接可以与手机通信。

相比苹果前不久推出的Homekit来说，Google的Brillo算是IOT行业第一个大佬级操作系统，也意在迅速占领这块苹果还没有染指的领域。而Weave可以看做是Google版的Homekit。

不过到年底之前，Brillo和Weave还不会有任何面向开发者或者用户的实际产品或者demo。所以估计起码得到2016年才会出来Brillo的第一批产品，不过也有开发者猜想，Nest产品的下一个版本可能就会装上这两样新技术。毕竟Google花大价钱收购的Nest可以很好的作为IOT行业的踏脚板。

Service的角色

Posted on 2015-01-08   |   In Android   |  

看到一篇关于描述Android里Service起到角色的帖子，感觉说的比较到位，翻译了一下。
原文地址：https://commonsware.com/blog/2014/07/27/role-services.html

直接给出结论：只有在Service真正正在给用户带来价值的时候才使用Service。
这句话我说的很多，因为这是我觉得最合适能表达什么时候该用Service的一句话。但是简略的几句话还是没法直观的说清楚这个问题。
首先，一个Service对操作系统来说应该像是一个Flag一样简单的是或者否的概念，表面你是否正在给用户带来价值。如果是，那么Android系统需要终止进程（比如内存不够）的时候就应该去寻找其他目标，而不是你的Service所在线程。第二，Service提供了一系列实现了IPC（跨进程通信）的API：一种命令模式（用Intent和startService()），一种绑定模式（用bindService()）。
当我们在单个进程内部用这两种通信模式时，主要的目的是就是为了让系统知道我们的Service确实在给用户带来价值。所以当Service没有给用户带来实际作用的时候，我们就应该停止Service。
所以当我们运行一个Service的时候，实际上我们在告诉Android系统当这个Service正在给用户带来价值的时候（后台数据下载，音乐播放等等），用户认为我们的进程比其他在竞争系统资源的线程更重要，所以用户希望系统要强关线程的时候关其他的而不是我们的。但是问题来了，当很多Service都霸占着线程的时候，就会影响系统的多任务，因为越多这种系统不该杀的线程存在，越多普通线程就容易被干掉。用户就会感觉到有些没用过多久的应用怎么就被系统干掉了，比如新闻类应用，本来缓存在一个新闻页然后切出去看了会微信，结果新闻类应用被干掉了，重新进来之后又得重新从首页导航加载。尤其是当应用本身没有处理好被系统强杀线程时候的操作（比如没有处理onSaveInstanceState()）。
这个问题的关键在于开发者没有以用户的角度来考虑，而是以他们自己的角度考虑。认为对他们重要的东西就也会对用户重要，实际上有的时候正好相反。开发者想早点干完活回家而不是清理蛋疼的Service代码，但是用户想要的是多任务运行顺利。
所以，再强调一遍，只有真正在给用户带来价值的时候才需要用Service。
在这里“价值”其实不太好定义，就像“美丽”一样，是比较主观的东西。而且用户才是那个最终使用的人，所以用户才应该是来定义这个“价值”的人。所以必须让用户感觉到你的Service带来的价值足够大，以至于可以接受多任务系统运行差一点。比如用户会到愤怒的小鸟页面的时候得从头开始，而不是回到保存的地方，因为Android因为用户接电话而内存不够，终止了小鸟的线程而不是你的线程。
“正在带来价值”指的事Service“此刻”正在带来价值，而不是“几分钟之内”这种模糊的概念，也不是“某时刻”。就是此刻。而且这个价值的接受者必须是用户。
这里就来举几个例子：
音乐播放应用正在播放音乐是一种正在传递价值给用户。如果该进程被杀，用户会立刻察觉到音乐停止，所以这种情况下是应该使用Service的。实际使用中，这种情况甚至还一般会加上startForeground()来更加提升进程的优先级。
让Service开着来保持线程中的数据而不是在重入时重新加载数据，这种情况就没有实际给用户带来价值。你的线程不会永远运行，所以更应该处理好线程重启和数据重载。对用户来说有用的数据被加载进来，但是并不需要一个长时间运行的Service。
用Service来下载大文件是在实时给用户带来价值。这个例子非常适合使用IntentService，因为你不仅需要Service，还需要一个后台线程来做网络和磁盘IO操作，并且下载完成之后Service会自己关掉。这样就可以保证Service对用户有用的时候运行，没用的时候就停止。
用Service来做定时性的工作基本上没有给用户实时带来价值。虽然定时性工作对用户有价值，但是看着定时器到达那个时间点的过程并没有什么意义。这种情况下更应该用AlarmManager，或者Android5.0新引进的JobScheduler。
用Service来获取一个应用的总体控制，这个也没有给用户带来什么实时的价值。这种情况更适合用一个轻量级的单例。
你觉得好像需要一个Service，但都还没有想好什么时候应该停止Service，就这么让它运行了。。这个更没有给用户带来任何价值。不要看心情来决定是不是要Service，而是确实在给用户带来价值。
一般以下几种情况我们会考虑用相应的Android组件。
· 当你想在屏幕上显示一些东西的时候用Activity。
· 当你想跟其他应用分享数据的时候用Content Provider。
· 当你要回应一个第三方广播的时候用Broadcast Receiver。
而Service是一个类似“瑞士军刀”一样的存在，可以用在很多的场景中。所以有的时候很难判断是不是要用Service。如果你的Service没有保持线程优先级的作用，使用Service本身就会是个很麻烦的问题。Service是会影响用户体验的，所以开发者一定要想清楚为什么要用Service，什么时候开始运行，什么时候停止运行。
所以还是回到最简单的那句话：只有在此刻正在给用户带来价值的情况要用Service。

关于Java中ArrayList的几个陷阱

Posted on 2013-10-26   |   In Java   |  

最近编了一些跟算法有关的东西，碰到了几个很奇怪的问题。找了一些资料和研究了下Java集合类和ArrayList的源码才彻底搞明白。

  • ArrayList add的时候传进的其实是引用，而并非值的拷。先看一个例子
public static void main(String[] args){
    ArrayList<ArrayList<String>> result = new ArrayList<ArrayList<String>>();
    ArrayList<String> values = new ArrayList<String>();

    values.add("a");
    values.add("b");
    values.add("c");
    result.add(values);
    values.clear();
    values.add("d");
    values.add("e");
    values.add("f");
    result.add(values);

    System.out.println(result.toString());
}

输出结果是：[[d, e, f], [d, e, f]]。而不是我们想的按照字面上的意思输出[[a, b, c], [d, e, f]]。
因为实际上在result.add(values)的时候传的是values这个list的引用，而不是整个list的拷贝。这样就导致其实下面在对values做处理的时候，已经加进result里的values也在跟着变化。
所以为了避免出现这种问题，改写成下面这样会让避免出现这个问题。

public static void main(String[] args){
    ArrayList<ArrayList<String>> result = new ArrayList<ArrayList<String>>();
    ArrayList<String> values = new ArrayList<String>();

    values.add("a");
    values.add("b");
    values.add("c");
    result.add(values);
    values = new ArrayList<String>();
    values.add("d");
    values.add("e");
    values.add("f");
    result.add(values);

    System.out.println(result.toString());
}
  • ArrayList在循环的时候删除元素。
public static void main(String[] args){

    ArrayList<String> values = new ArrayList<String>();
    values.add("0");
    values.add("1");
    values.add("2");
    values.add("3");
    values.add("4");

    for(int i=0;i<values.size();i++){
        values.remove(i);
    }
    System.out.println(values.toString());
}

结果为: [1, 3]。这是因为在删除操作之后，整个list的index本身就也产生了变化，所以继续做删除的时候就已经不是一开始那个list的index，这样就产生了错误的结果。
要是写成这样呢？

Iterator<String> iterator = values.iterator();
while(iterator.hasNext()){
    String element = iterator.next();
    values.remove(element);
}

结果出现了ConcurrentModificationException，其实这是因为在Iterator做任何操作的时候都会做checkForComodification，其实就是检查是否list长度有改动，如果有改动并且没有通知到iterator进行更新，就会抛出ConcurrentModificationException。
正确的写法是使用Iterator本身提供的remove方法，这样的话每次删除元素更新的列表长度都会被Iterator记录下来，并始终保持一致，这样就不会再出现ConcurrentModificationException。

 Iterator<String> iterator = values.iterator();
while(iterator.hasNext()){
    iterator.next();
    iterator.remove();
}

SLIP协议

Posted on 2013-08-07   |   In 通信   |  

SLIP协议（Serial Line Interenet Prococol)串行网络协议。RFC1055

早期互联网时代，还没有发明以太网双绞线这种可以传输查分信号的东西，传输网络信号的介质有限。所以最早的Internet是通过串口来传输的。从今天的角度来看，当年的互联网真的是慢的可怜，工业界用的的串口速度很多都在115200字节每秒。但是这样的速度在当年已经可以实现一些最最基本的基于文本的互联网。

要通过串口实现上网，一个很重要的问题是串口得有能力区分包。因为串口本身是一个二进制字节流，并不提供任何协议，基于分包的互联网在串口上传输，怎么在接收端知道收到的是一个一个包呢。SLIP协议就是用来解决这个问题的。

首先为了实现分包，每个包都必须要求有一个包头，可以让接收端识别这个包头，才知道要开始接收的是一个包。而在SLIP协议里就用了0xC0这个字节专门规定为包头（也同时为包尾）。

那么问题又来了，要是规定了这个字节当包头，那么发送端就在每个包前面加上这个字节，但是因为数据流里面是任何数据都有可能存在的，要是数据流里本身就包含了0xC0的话，接收端就会分错包。SLIP里的解决办法是发送端只要碰到0xC0就把它转换成两个字节0xDB+0xDC，并且把0xDB也定义成特殊字符，否则的话碰到0xDB又会当做数据帧处理，每当碰到0xDB就替换成0xDB+0xDD。做过这样的处理之后在接收端只要收到0xC0就肯定知道是包头，然后收到0xDB的时候要等到下一个才知道是什么数据，再把两个数据一起还原成0xC0或者0xDB。

RFC1055中已经给出C的简单实现。这里给出Java的实现版本。

private static final int END = 0xC0;
private static final int ESC = 0xDB;
private static final int END_ESC = 0xDC;
private static final int ESC_ESC = 0xDD;

/**
 * Send a packet
 * 
 * @param packet
 */
private static void send(byte[] packet) {

    sendChar(END);

    for (int i = 0; i < packet.length; i++) {
        switch (packet[i] & 0xFF) {
        case END:
            sendChar(ESC);
            sendChar(END_ESC);
            break;
        case ESC:
            sendChar(ESC);
            sendChar(ESC_ESC);
            break;
        default:
            sendChar(packet[i]);
            break;
        }
    }
}

/**
 * Receive a Packet
 * @return packet bytes 
 * @throws Exception
 */
private static ArrayList<Integer> receive() throws Exception {
    ArrayList<Integer> recvData = new ArrayList<Integer>();


    boolean endFlag = false;        
    while(true){
        int data = getChar();

        switch(data){
        case END:
            if(endFlag){
                //Packet END found
                endFlag = false;
                return recvData;
            }
            else{
                endFlag = true;
                break;
            }

        case ESC:
            int c = getChar();
            switch(c){
            case END_ESC:
                recvData.add(END);
                break;
            case ESC_ESC:
                recvData.add(ESC);
                break;
            default:
                //After ESC, no conventional byte detected
                throw new Exception("Receive Error");
            }
        default:
            if(endFlag){
                recvData.add(data);
            }
            break;    
        }
    }

}

Android WiFi调试

Posted on 2013-02-17   |   In Android   |  

有的时候调试Android的程序，不方便一直插着USB线调试，其实Android还提供一种基于wifi的调试方法。

首先先保证Adroid手机和电脑在同一个WiFi局域网内，不然基于WiFi调试的前提都没有达到。然后手机的USB debug模式要已经打开。

然后Android手机还是用USB插在电脑上，打开一个命令行，找到Android SDK文件夹里面的platform-tools里的adb.exe，先执行adb devices看手机是否连上了adb。

然后执行adb tcpip 5555。这样就打开了手机上的5555端口用来发送手机的log。

Android手机已经和PC在同一个局域网中了，找到Android里的WiFi设置-高级里面，可以看到当前IP地址。

最后在PC的命令行里执行adb connect 192.168.1.XXX(Android的IP地址)。就可以看到手机和adb通过WiFi连在了一起，这个时候即使拔掉USB，在eclipse上也还是可以看到logcat在继续输出Android上的信息。
